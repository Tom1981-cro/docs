Veritome enforces rate limits on all API endpoints to ensure fair usage and platform stability. Limits are tracked **per API key** using a sliding-window algorithm powered by Upstash Redis. When your key exceeds its limit within the current window, the API returns `429 Too Many Requests` until the window resets.

## Default limits

Rate limits vary by plan and by endpoint category. Sensitive actions such as creating or rotating API keys are subject to tighter per-user limits (10 requests per minute) regardless of plan. General API endpoints use the limits below:

| Plan | Requests per minute |
|------|-------------------|
| Growth | 100 |
| Core | 300 |
| Enterprise | Custom (contact us) |

<Note>
  Limits may be adjusted over time. Contact [support@veritome.eu](mailto:support@veritome.eu) if your integration requires a higher quota.
</Note>

## 429 response format

When your key is rate-limited, the API responds with:

- **HTTP status:** `429 Too Many Requests`
- **`Retry-After` header:** the number of seconds to wait before retrying
- **Response body:**

```json
{
  "error": "Too many requests. Please slow down.",
  "retryAfter": 14
}
```

The `retryAfter` value is calculated from the exact moment the current window resets, so it is always the minimum wait required — not a rounded estimate.

## Handle 429 responses with exponential backoff

Rather than retrying immediately (which would continue to consume quota), implement exponential backoff with jitter. The following examples show a robust retry helper:

### TypeScript

```typescript
async function fetchWithRetry(
  url: string,
  options: RequestInit,
  maxRetries = 4
): Promise<Response> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status !== 429) {
      return response;
    }

    if (attempt === maxRetries) {
      throw new Error(`Rate limit exceeded after ${maxRetries} retries`);
    }

    const retryAfter = parseInt(response.headers.get("Retry-After") ?? "1", 10);
    // Exponential backoff: retryAfter + jitter
    const backoff = retryAfter * 1000 + Math.random() * 500;
    console.warn(`Rate limited. Retrying in ${(backoff / 1000).toFixed(1)}s…`);
    await new Promise((resolve) => setTimeout(resolve, backoff));
  }

  throw new Error("Unreachable");
}

// Usage
const response = await fetchWithRetry(
  "https://veritome.eu/api/systems",
  {
    headers: { Authorization: `Bearer ${process.env.VERITOME_API_KEY}` },
  }
);
```

### Python

```python
import time
import random
import requests

def fetch_with_retry(url: str, headers: dict, max_retries: int = 4) -> requests.Response:
    for attempt in range(max_retries + 1):
        response = requests.get(url, headers=headers)

        if response.status_code != 429:
            return response

        if attempt == max_retries:
            raise Exception(f"Rate limit exceeded after {max_retries} retries")

        retry_after = int(response.headers.get("Retry-After", 1))
        # Exponential backoff with jitter
        backoff = retry_after + random.uniform(0, 0.5)
        print(f"Rate limited. Retrying in {backoff:.1f}s…")
        time.sleep(backoff)

# Usage
API_KEY = "vrt_live_..."
response = fetch_with_retry(
    "https://veritome.eu/api/systems",
    headers={"Authorization": f"Bearer {API_KEY}"}
)
data = response.json()
```

## Reduce API calls with webhooks

Polling endpoints repeatedly to detect state changes (e.g. waiting for an obligation to be approved, or checking whether a dossier has been generated) is one of the most common causes of rate-limit exhaustion. Instead, configure a [webhook endpoint](/api/webhooks) so that Veritome pushes events to your server the moment they occur.

<Tip>
  A single webhook subscription for `obligation.approved` or `dossier.generated` eliminates the need for any polling loop and reduces your API call volume to near zero for those workflows.
</Tip>

Webhooks are available on the Growth plan and above from **Settings → API → Webhooks**.
